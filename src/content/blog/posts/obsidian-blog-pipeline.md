# 블로그가 옵시디언 안으로 들어오기까지

블로그를 [포트폴리오 안에](/blog/portfolio-bear-blog) 넣고 나서 바로 다음 문제가 보였다. 글을 어디서 쓰느냐는 문제다.

처음 구조는 단순했다. 레포 안에 마크다운 파일을 만들고, frontmatter를 채우고, 커밋한다. 동작은 한다. 근데 나는 이미 모든 글을 옵시디언에 쓰고 있다. 독후감도, 회고도, 아이디어 메모도 전부 vault에 있다. 블로그 글만 레포에 따로 쓴다는 건 ==글 쓰는 곳이 두 군데가 된다는 뜻==이고, 경험상 그런 구조는 반드시 한쪽이 죽는다. 죽는 쪽은 항상 더 귀찮은 쪽이다.

> 도구가 두 개면 결국 둘 다 안 쓰게 되고, 도구가 하나면 그 도구의 한계까지는 쓰게 된다.

그래서 방향을 정했다. **옵시디언이 원본이고, 레포는 빌드 산출물이다.**

---

## 구조

vault에 `blog/` 폴더를 만들고, 그 아래 폴더 이름이 곧 블로그의 갈래가 되게 했다.

```
08 Me/blog/
├── 설계와 구현/      ← 폴더명 = 카테고리
├── 생각과 질문/
├── 읽기와 보기/
├── 견문과 모임/
└── 해의 회고/
```

동기화 스크립트가 이 폴더를 훑어서 레포의 `src/content/blog/posts/`를 통째로 다시 만든다. 핵심 결정은 세 가지였다.

| 결정 | 선택 | 이유 |
| --- | --- | --- |
| 발행 기준 | frontmatter `publish: true` | 초고가 실수로 나가는 게 최악의 사고라서 |
| 동기화 방식 | 전량 재생성 | 부분 갱신은 삭제·이동 추적이 필요해져서 복잡도만 늘어남 |
| 카테고리 | 폴더 위치 | 속성으로 두면 폴더랑 어긋나는 순간이 반드시 옴 |

> [!warning] 전량 재생성의 대가
> 레포 쪽 `posts/`를 직접 고치면 다음 동기화 때 흔적도 없이 사라진다. 글 수정은 무조건 vault에서. 원본이 하나라는 건 이런 뜻이다.

## 옵시디언 문법은 옵시디언 밖에서 죽는다

진짜 일은 변환이었다. 옵시디언은 표준 마크다운이 아니다. `![[이미지]]`, `[[내부 링크]]`, `%%주석%%`, 콜아웃 — vault 밖으로 나가는 순간 전부 깨진다.

이미지가 제일 까다로웠다. 옵시디언 첨부는 vault 어딘가에 흩어져 있고, 노트는 파일명만 안다. 그래서 동기화 때 vault 전체를 한 번 인덱싱한다.

```ts
// 첨부는 vault 어디에 있어도 basename으로 찾는다 (옵시디언 기본 동작과 동일)
async function buildAssetIndex() {
  const index = new Map();
  const files = await walk(VAULT_ROOT, { skip: new Set([".obsidian", ".trash"]) });
  for (const file of files) {
    if (!ASSET_EXTENSIONS.has(path.extname(file).toLowerCase())) continue;
    const base = path.basename(file);
    if (!index.has(base)) index.set(base, file);
  }
  return index;
}
```

내부 링크는 한 가지 규칙으로 정리했다. 링크 대상도 발행된 글이면 블로그 링크로 바꾸고, 아니면 그냥 텍스트로 푼다. 비공개 노트로 가는 죽은 링크를 만드는 것보다 링크를 포기하는 쪽이 낫다.

> [!tip] 읽기 시간 계산
> 글머리에 붙는 "N분 읽기"는 한국어 기준으로 대충 이렇게 잡았다.
>
> $$t = \lceil \frac{\text{공백 제외 글자 수}}{500} \rceil$$
>
> 분당 500자는 보수적인 값인데, 기술 글은 코드 읽는 시간이 있어서 이 정도가 체감에 맞다.

## 발행은 명령어 하나

쓰는 흐름이 길면 안 쓴다. 그래서 발행을 셸 한 줄로 줄였다.

```bash
blogpub   # sync → 변경분 커밋 → push → 자동 배포
```

내부는 별거 없다. 동기화 돌리고, 바뀐 글만 커밋하고, 푸시한다. 푸시가 곧 배포라서[^1] 옵시디언에서 글을 닫고 터미널에 여덟 글자 치면 끝난다.

> [!note] 지금 이 글이 증명이다
> 이 글 자체가 vault의 `설계와 구현/` 폴더에서 `blogpub` 한 번으로 발행됐다. 파이프라인 글을 그 파이프라인으로 발행하는 것만큼 확실한 테스트는 없다.

결국 이 작업의 본질은 기능이 아니라 마찰 제거다. 글쓰기를 막는 건 능력이 아니라 귀찮음이고, 귀찮음은 구조로만 이긴다. 이제 변명거리가 하나 줄었으니, 남은 건 쓰는 일뿐이다.

[^1]: GitHub Actions가 main 푸시를 받아 홈서버에서 도커를 다시 올린다. 이 경로도 최근에 Tailscale로 갈아엎었는데, 그 얘기는 따로 쓸 만큼 길다.
